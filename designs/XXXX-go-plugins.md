# Design: CUE Go Plugins

NOTE: this document is a work in progress. It is not
yet intending to adhere to our conventions for public design proposals.

## Overview

CUE external funcs associated directly with Go code. We want to build a
binary that contains the Go code associated with a CUE package/module.

### Envisaged flow

- A publisher P wishes to publish a CUE module containing a package
  with co-located Go and CUE.

- P can use `cue plugin init-go` to initialize such a package, which
  will inspect Go functions in the current package to create an initial
  CUE package with `@extern(go)` attributes.

- P edits the initial file to their taste, perhaps pulling in functions
  from other packages too, or renaming functions.

- P runs `cue plugin generate-go` to generate a CUE file (for func
  signatures) and a Go file (for imported packages).

- P publishes the module to a registry.

- A consumer C wishes to use the published module.

- C imports the published module and runs `cue mod tidy`
  to add it to their dependencies.

- C runs `cue plugin build` which looks at the current module and its dependencies to
  determine what plugins are needed. It generates a wrapper main module that
  defines a `main` function that is like the entry point for the regular `cue`
  binary but also includes all the necessary plugin values implied by the
  module's dependencies. This generated code and its associated `go.mod` file
  (defining the Go dependencies for all plugin code) are created inside the
  `cue.mod/plugin/` directory. It builds and caches a cue binary that has all
  the functionality of a regular cue binary, built from the generated main
  module code.

- Once a plugin executable is built, C can run the regular cue binary,
  which detects (by looking in the `cue.mod/plugin/` directory) that a
  plugin executable is needed, and re-executes that to do all the work.

- When the plugin executable executes CUE code that requires a plugin value,
  it can look at the plugins registered in it and inject the correct value (or
  fail if that plugin isn't available).

## Example published CUE/Go module

Below is a fully worked example of a CUE module co-located with Go,
ready for publishing. The `_gen.*` files would be generated
automatically by calling `cue plugin generate-go`.

is exactly that of Go — we use the Go parser logic and AST to parse
it.

package sprig

// Substr returns the substring of s from byte index start to end.
func Substr(start, end int, s string) string {
	if start < 0 {

## Design details

### `cue plugin init-go`

Scaffolds an initial CUE file with `@extern(go)` declarations from the Go source
code in the current directory.

**Preconditions**: The current directory must be within a CUE module and must
contain a Go package.

**Operation**:

1. Verify a CUE module exists (search upward for `cue.mod/`).
2. Load the Go package in the current directory using
   `golang.org/x/tools/go/packages`.
3. Find all exported functions whose parameter and return types are
   CUE-representable (basic types, structs of basic types, slices, maps,
   pointers). Functions returning `(T, error)` are eligible — the error
   is handled by the `cue.NewPureFuncN` wrapper. Functions must have
   1–10 parameters.
4. Generate a `.cue` file containing:
   - An `@extern(go)` file-level attribute (no `import=(...)` needed when
     all references are to the local Go package).
   - A `package` clause.
   - One field per discovered function: `FuncName: _ @go()`.
5. Write the file to `<pkgname>.cue` (or a user-specified name via `--output`).

**Flags**:

- `--output` / `-o`: Output filename (default: `<pkgname>.cue`).
- `--package` / `-p`: Override the CUE package name (default: Go package name).
- `--force`: Overwrite an existing file (by default, refuses to overwrite).

**Error cases**: No Go package found; not in a CUE module.

If there are no compatible exported functions, write out a file with a `@extern(go)`
skeleton and a brief explanation of what to do next.

### `cue plugin generate-go`

From existing `@extern(go)` / `@go(...)` declarations, generates two files
per source: a CUE file with function signatures and a Go file with blank imports.

**Operation**:

1. Load the CUE package in the current directory.
2. Extract all `@extern(go)` files and their `@go(...)` references.
3. For each source file containing `@extern(go)`, generate blank import Go
   files first (so that `go mod tidy` can see all referenced packages).
4. Run `go mod tidy` to ensure all referenced packages are available.
5. Load Go type information for all referenced packages using
   `golang.org/x/tools/go/packages`.
6. For each reference, map the Go function signature to CUE types.
7. Generate `<stem>_signatures_gen.cue`: one field per function with a doc
   comment showing the signature (e.g., `// func(string) -> string`) and
   a `_` body. Go doc comments are propagated to the CUE file if the CUE
   field does not already have a doc comment. The `package` clause matches
   the source package.
8. Generate `<stem>_imports_gen.go`: a Go file with a `//go:build cueplugin`
   build tag, package name `cueplugin_imports`, and blank imports
   (`_ "pkg/path"`) for every externally referenced Go package. This
   ensures `go mod tidy` retains the dependencies in `go.mod`.

The `<stem>` is derived from the CUE source file containing the
`@extern(go)` declaration (e.g., `sprig.cue` → `sprig_signatures_gen.cue`
and `sprig_imports_gen.go`). No `_imports_gen.go` file is generated when
all references are local.

**Regeneration**: Always overwrites `_gen.*` files — they carry a
"DO NOT EDIT" header.

**Error cases**: Referenced Go function not found; function has unsupported
parameter types; Go package fails to compile.

### `cue plugin build`

Generates a wrapper main module in `cue.mod/plugin/` and builds a cached
plugin binary that extends the regular `cue` binary with all required Go
plugin values.

**Operation**:

1. Find the CUE module root and load all CUE instances (`./...`).
2. Walk the full import graph of all instances and call
   `goplugin.References()` to collect all Go references. If none
   are found, report "no plugins needed" and exit.
3. For each reference, find the `go.mod` file by searching upward
   from the CUE instance's directory (stopping at the CUE module root).
   When `GoPackage` is `"."`, resolve the full Go package path from the
   Go module path plus the relative directory.
4. For each Go package, determine which Go module provides it:
   - If it appears in the `go.mod` requirements, use that version directly.
   - If it is local to the `go.mod`'s own module (and the CUE module has
     VCS commit metadata in the registry), use the commit hash as a
     pseudo-version.
   - Otherwise, fall back to a `v0.0.0` version with a `replace` directive
     pointing to the local directory.
5. Generate `cue.mod/plugin/go.mod`:
   - Module name `cue_plugin_gen`.
   - A `go` directive matching the current Go version (major.minor).
   - A `require` directive for each referenced Go module and for
     `cuelang.org/go` (at the version matching the running `cue` binary,
     obtained via `cueversion.ModuleVersion()`).
   - A `replace` directive when a referenced Go module is local.
6. Generate an interim `cue.mod/plugin/main.go` with blank imports for
   all referenced packages plus the `cuelang.org/go` packages needed by the
   final code.
7. Run `go mod tidy` in `cue.mod/plugin/` to resolve transitive dependencies.
8. Load all referenced Go packages with type information using
   `golang.org/x/tools/go/packages`.
9. For each reference, look up the Go function, inspect its signature, and
   generate a typed wrapper expression using the appropriate
   `cue.NewPureFuncN` variant (N = number of parameters, 1–10).
   Functions that return `(T, error)` are used directly; functions
   returning only `T` get a wrapper that adds a `nil` error return.
10. Generate the final `cue.mod/plugin/main.go` from a template, containing:
    - Imports of all referenced Go packages (with generated aliases based
      on actual Go package names, disambiguated with numeric suffixes if
      needed).
    - A `map[goplugin.Reference]func() cue.Value` literal mapping every
      collected reference to its wrapper function.
    - A `main` function that calls
      `os.Exit(cmd.MainWithOptions(cuecontext.WithInjection(goplugin.Injection(refs))))`.
11. Generate `cue.mod/plugin/manifest.json` recording which CUE modules,
    instances, Go packages, and function names are referenced (see
    "Manifest format" below).
12. Build the binary: run `go build -trimpath` in `cue.mod/plugin/`,
    placing the output in `CUE_CACHE_DIR` at a path derived from a
    content hash (see "Cache invalidation and correctness" below).

**Error cases**: No plugins needed; Go compilation failure (Go compiler
errors are surfaced directly); `go.mod` not found for local references;
Go module version resolution failure; plugin reference to `cuelang.org/go`
itself (currently forbidden).

### The `@extern(go)` attribute

The `@extern(go)` attribute is a file-level attribute that marks a CUE
file as containing Go plugin references. It has an optional `import`
parameter that declares Go package imports using standard Go syntax:

```
@extern(go)
```

or with external imports:

```
@extern(go, import=(
    utils "github.com/Masterminds/goutils"
    "github.com/huandu/xstrings"
))
```

The `import=(...)` value is parsed using the Go parser — its contents
are exactly Go import syntax (with optional aliases). When no alias is
given, the default identifier is the last path element (with special
handling for `v2`+ major version suffixes, matching Go conventions).

### The `@go(...)` attribute

The `@go(...)` attribute is placed on individual fields and specifies
which Go function to bind. It has three forms:

- **`@go()`** — empty: uses the CUE field name as the Go function name
  in the local (co-located) Go package. The field `Upper: _ @go()` binds
  to the `Upper` function in the local Go package.

- **`@go(Name)`** — unqualified: uses the given name as the Go function
  name in the local Go package. The field `OptionalFunc: _ @go(OptFunc)`
  binds to the `OptFunc` function locally.

- **`@go(pkg.Name)`** — qualified: uses an import alias from the
  file-level `@extern(go, import=(...))` declaration to look up the Go
  package, then references the named function in that package. The field
  `Untitle: _ @go(utils.Uncapitalize)` references `Uncapitalize` in
  whichever package `utils` is aliased to.

The referenced Go function must be exported and must have 1–10 parameters
with CUE-representable types and return either `T` or `(T, error)`.

### Plugin go.mod structure

The generated plugin `go.mod` (`cue.mod/plugin/go.mod`) contains:

- Module name `cue_plugin_gen`.
- A `go` directive for the current Go toolchain version.
- A `require` directive for each Go module that provides plugin packages,
  plus `cuelang.org/go`.
- A `replace` directive pointing to the local directory when a referenced
  Go module is local (no published version available).

After initial generation, `go mod tidy` is run to let Go's MVS resolve
all transitive dependencies.

### Manifest format

The file `cue.mod/plugin/manifest.json` records all plugin references and
their resolution. It has the following structure (using the
`goplugin.Manifest` type):

```json
{
    "modules": {
        "<CUE module path>": {
            "instances": {
                "<CUE import path>": {
                    "colocatedGoPackage": "<Go import path>",
                    "goPackages": {
                        "<Go import path>": ["FuncName1", "FuncName2"]
                    }
                }
            }
        }
    },
    "goPackages": {
        "<Go import path>": {
            "path": "<Go module path>",
            "version": "<Go module version>",
            "dir": "<local directory (for replace directives)>"
        }
    }
}
```

The `colocatedGoPackage` field is present only when the instance has
`@go()` references that use `"."` (the local Go package). The
`goPackages` map within each instance lists every Go package referenced
and the function names used from it. The top-level `goPackages` map
records which Go module provides each referenced Go package, along with
the version or local directory.

The manifest is used by the `Checker` injection (see below) to resolve
references without needing to re-parse CUE files.

### Cache invalidation and correctness

#### Defining correctness

A "correct" cache means: **an evaluation produces the same result
regardless of whether any cached state is present or not.** That is, if
the user were to delete all cached state and re-run `cue plugin build`
followed by evaluation, the result would be identical to using the
existing cache.

Full correctness by this definition would require re-running the
equivalent of `cue plugin build` on every evaluation — walking all CUE
module dependencies, finding their `go.mod` files, resolving Go packages
via MVS, and building the binary — because even if the current binary
already contains every plugin reference needed for this evaluation, the
addition of a *new* CUE dependency could (via MVS) bump a Go module
version used by an existing plugin, changing that plugin's behavior.

This full walk is prohibitively expensive for every evaluation. We
therefore accept some degree of staleness in exchange for fast startup.
The remainder of this section characterises the assumptions we make, the
scenarios in which they break, and possible future improvements.

#### Two layers of caching

There are two distinct layers of cached state, each of which can become
stale independently:

1. **The generated source files in `cue.mod/plugin/`.** These are
   `go.mod`, `main.go`, and `manifest.json`, produced by
   `cue plugin build`. They are a materialised snapshot of the
   dependency resolution: which CUE modules contribute plugins, which Go
   packages they reference, which Go module versions MVS selected, and
   which functions are wrapped. These files are expected to be committed
   to version control, so they also serve as a record of what was built.
   They become stale when the inputs to `cue plugin build` change (CUE
   dependencies, local Go source, etc.) without the command being re-run.

2. **The compiled binary in `CUE_CACHE_DIR/plugin/`.** This is derived
   from the generated source files by `go build`. Its path is a SHA-256
   hash of `GOARCH`, `GOOS`, and the contents of the generated `go.mod`
   and `main.go`. It becomes stale when the generated files change
   (requiring a `go build`) or when the Go toolchain changes (producing
   a different binary from the same source).

At evaluation time, the regular `cue` binary checks whether
`cue.mod/plugin/` exists. If so, it hashes the generated files and
looks for the binary at the computed path. If found, it re-execs it. If
not found, it tells the user to run `cue plugin build`.

Note that the binary cache is self-correcting with respect to the
generated files: if someone edits `cue.mod/plugin/main.go` or re-runs
`cue plugin build`, the hash changes and the old binary is simply not
found, prompting a rebuild. The harder problem is staleness of the
generated files themselves, because nothing currently detects when they
no longer reflect the true state of the module's dependencies.

#### Assumptions and where they hold

The PoC relies on three key assumptions:

1. **Versioned CUE module dependencies are immutable.** A given CUE
   module version always has the same content, and therefore the same
   `go.mod`, the same `@extern(go)` declarations, and the same Go
   source code. This is the same immutability assumption that the CUE
   module cache already makes (see `CUE_CACHE_DIR` in
   `cue help environment`).

2. **The generated `cue.mod/plugin/` files are up to date.** The user
   has run `cue plugin build` since the last change to the module's
   dependency set, and has committed the results.

3. **The Go download cache remains unchanged. The Go build cache
functions on the file contents, not the module names, but unlike
`go run`, we do not read and hash all the Go dependency code
to determine the computed binary path.

Under these assumptions, the generated `go.mod` and `main.go` fully
determine the binary's behavior for versioned dependencies: the `go.mod`
pins exact Go module versions (after `go mod tidy`), and `main.go`
enumerates exactly which functions are wrapped. Since the binary hash
covers both files, a matching binary is correct.

#### Scenarios that break correctness

The following scenarios can cause evaluation to produce different results
than it would with a fresh `cue plugin build`:

**1. CUE dependency set changes without re-running `cue plugin build`.**
The user adds, removes, or upgrades a CUE module dependency, but does
not rebuild the plugin. The generated `cue.mod/plugin/` files are now
stale.

- *If the new dependency introduces new plugin references:* evaluation
  fails with "no Go plugin registered", because the binary's reference
  map doesn't include them. This is a correct failure — it forces the
  user to rebuild.
- *If the new dependency does not introduce new plugin references but
  its transitive Go dependencies (via a shared Go module) would bump a
  version used by an existing plugin:* the generated `go.mod` still pins
  the old Go dependency versions. The binary silently uses the old
  versions. This is a **silent correctness violation** — evaluation
  succeeds but may produce different results than a fresh build. In
  practice, Go's compatibility promise and semver discipline make
  behavioral changes across minor versions rare, but it is possible.

**2. Local Go source changes (when the main module defines plugins).**
The generated `cue.mod/plugin/` files reference the local Go package by
import path but do not capture its source contents. If the user edits a
local Go function's implementation, neither the generated files nor the
binary hash changes, so the cached binary is reused with the old
behavior. The user must re-run `cue plugin build` manually.

**3. Go toolchain version changes.** The binary hash includes `GOARCH`
and `GOOS` but not the Go compiler version. A different Go toolchain
could produce a binary with different behavior (e.g. different
floating-point results, stdlib behavior changes). Additionally, the
generated `go.mod` pins a `go` directive version at generation time;
building with a different toolchain version doesn't change the generated
files. This is unlikely to matter in practice but is technically a
correctness gap.

**4. `CUE_REGISTRY` changes.** Neither layer of cache depends on the
registry configuration. If the user switches registries, a CUE module
version that resolves to different content could exist at the same
version string. The CUE module cache has the same limitation.

#### Justification for the trade-offs

Scenarios 1 and 2 are the most practically relevant. Both require the
user to run `cue plugin build` to recover correctness, which matches
the existing user expectation: `cue plugin build` is an explicit step,
analogous to running `go build` after changing Go source.

The silent case in scenario 1 (MVS version bumps from new unrelated
dependencies) is the most concerning, but is bounded by two factors:
Go's compatibility promise limits behavioral changes across minor
versions, and the scenario requires a specific combination of events
(adding a CUE dependency that shares a transitive Go dependency at a
higher version).

Scenarios 3 and 4 are edge cases shared with the broader Go and CUE
ecosystems respectively.

#### Possible future improvements

Several approaches could improve correctness without requiring a full
dependency walk on every evaluation:

- **Manifest-based staleness check.** The manifest records which CUE
  modules contribute plugins. At evaluation time, compare this set
  against the current `cue.mod/module.cue` dependency list. If they
  differ, warn (or error) that `cue plugin build` may be needed. This
  is a cheap file read and comparison — no network, no Go toolchain
  invocation — and catches scenario 1 (both the hard-failure and silent
  cases). This addresses staleness of the generated files, which is the
  more important of the two cache layers.

- **Including local source in the hash.** For local references,
  include checksums of the relevant `.go` files in the binary cache key.
  The set of relevant files can be determined via `go list` (which
  correctly handles build constraints and `//go:build` directives). This
  addresses scenario 2 but adds the cost of a `go list` invocation to
  every evaluation startup. Alternatively, checksum the files only when
  the manifest indicates local references exist.

- **Go build ID comparison.** After building, store the Go build ID
  (`go tool buildid`) alongside the binary. On subsequent evaluations,
  re-run `go tool buildid` on the generated source and compare. This
  captures all Go-level inputs (including toolchain version) but
  requires invoking the Go toolchain, making it more expensive than a
  pure file hash.

- **Automatic rebuild.** Rather than erroring when staleness is
  detected, `cue eval` could automatically run the equivalent of
  `cue plugin build`. This trades latency for correctness but could
  surprise users (unexpected network access, build failures during
  evaluation). It could be gated behind a flag or environment variable.

- **Two-tier check.** Use the cheap binary hash for the fast path; if
  it matches, additionally run the manifest-based staleness check
  against `cue.mod/module.cue`. If that fails, warn but still use the
  cached binary (or error, depending on a strictness setting). This
  provides a good balance: the cheap check handles the binary layer,
  and the manifest comparison catches the more important generated-files
  layer, all without invoking the Go toolchain.

### go.mod placement

The `go.mod` file for a CUE module's Go plugin code must be at or beneath the
CUE module root (the parent directory of `cue.mod/`). It cannot be outside of
that, because it would then be outside the CUE module contents and we wouldn't
be able to determine the Go module name that the Go code belongs to.

The `go.mod` is located by searching upward from the directory containing the
Go plugin source code, stopping at the CUE module root. It is an error if
no `go.mod` is found within the CUE module root.

This is user-managed, maintained like any `go.mod` file (typically with
`go mod tidy`). A `go.mod` is only needed in CUE modules that *define* plugins,
not in modules that merely consume them.

### The `goplugin.Reference` type

A `goplugin.Reference` identifies a single Go function referenced from CUE
code. It is used as a map key in the generated plugin code. Its fields are:

- `GoPackage` (string): The Go package path being referred to. This is `"."`
  when referring to the co-located Go package, in which case `CUEInstance`
  is used for lookup instead.
- `CUEInstance` (string): The CUE import path of the instance, set only
  when `GoPackage` is `"."`.
- `CUEModuleVersion` (string): The `module@version` string of the CUE module
  containing the reference. Empty when the reference is from the main module.
- `Name` (string): The exported Go function name.

### The `goplugin.Injection` function

`goplugin.Injection` creates a `cuecontext.Injection` from a reference map:

```go
func Injection(refMap map[Reference]func() cue.Value) cuecontext.Injection
```

This is used in the generated `main.go` to register all plugin functions.
The injection implements the `"go"` kind and, for each `@go(...)` attribute
encountered during evaluation, looks up the corresponding `Reference` in
the map and returns the CUE value produced by the associated function.

### The `goplugin.Checker` function

`goplugin.Checker` creates a `cuecontext.Injection` that validates plugin
references against a `Manifest` without providing actual implementations:

```go
func Checker(m *Manifest, check func(ref ResolvedReference) error) cuecontext.Injection
```

For each `@go(...)` attribute encountered, the Checker resolves the reference
using the manifest and calls the check function. If the check returns nil,
the injected value is `_` (top); if it returns an error, the error is
propagated. This is used by the plugin binary to validate that all plugin
references are satisfiable.

### Cross-module Go dependencies

The rule for Go dependencies between plugin packages: **Go dependencies are
resolved purely by Go's module system.** The CUE module system determines
*which* plugin packages exist; Go's module system resolves their Go-level
dependencies.

What is **forbidden** is Go code in one CUE module directly depending on Go
code that is *co-located with* (part of) another CUE module. Shared Go utility
libraries that happen to be used by multiple plugins are fine — the restriction
only applies when the Go module is directly part of another CUE module's source
tree.

Circular dependencies should also be forbidden: CUE module A's Go code
importing the Go code of CUE module B while B's CUE code depends on A.

### Mapping from CUE module to Go module for external dependencies

For CUE modules that provide Go plugins, we look at the CUE module source and
find a `go.mod` file within it. This gives us the Go module name (but not its
version).

To determine the actual Go module version to depend on:

1. If the Go package appears in the `go.mod` requirements of the CUE module's
   co-located Go module, use that version directly.
2. Otherwise, if the Go package is local to the `go.mod`'s own module, attempt
   to find a VCS commit from the CUE module metadata (as stored by
   `modregistry.Client`). The raw commit hash is used directly — `go mod tidy`
   resolves it to a proper pseudo-version.
3. If no VCS metadata is available, fall back to a `v0.0.0` version with a
   `replace` directive pointing to the local directory.

It is the responsibility of the publisher of a CUE module with Go plugins to
ensure their Go code is amenable to this mapping. References to Go packages
that are replaced in the publisher's `go.mod` are currently an error.

### Re-exec strategy

If `cue.mod/plugin/` exists, the regular `cue` binary treats this as a
signal that a plugin binary is needed. It hashes the `go.mod` and `main.go`
files in that directory to determine the expected binary path in
`CUE_CACHE_DIR/plugin/`, and re-execs it. The plugin binary is a superset
of the regular binary, so this is always safe. The only cost is the exec
overhead, which is negligible.

The detection works as follows: when `MainWithOptions` is called with no
options (i.e. the regular `cue` binary), the `runningInPlugin` flag is false.
If any command's `RunE` encounters a `needPluginError` (from checking for
the `cue.mod/plugin/` directory), `Main` catches it and calls `execPlugin`,
which locates and re-execs the cached binary. If the cached binary is
missing, it prints an actionable error telling the user to run
`cue plugin build`.

### Versioning and compatibility

The plugin binary embeds a specific version of the CUE evaluator (from
`cuelang.org/go`). If the user upgrades their `cue` CLI, the cached plugin
binary still has the old evaluator. This could cause evaluation differences or
subtle bugs from version skew.

The plugin binary's CUE evaluator version should be recorded in its metadata.
When the regular `cue` binary detects a plugin, it should check version
compatibility and refuse to re-exec (or warn and rebuild) if versions diverge.

To seed the correct version, `cue plugin build` determines the `cuelang.org/go`
version from `cueversion.ModuleVersion()`. If this is a valid semver version,
it is used directly. If not (e.g. when the `cue` binary is built from a local
fork or an unreleased commit), the `CUE_PLUGIN_CUE_REPLACE` environment
variable must be set to a local path, which is used as a `replace` directive.

### Relationship between `@extern(go)` and `@extern(inject)`

`@extern(go)` and `@extern(inject)` are independent but related mechanisms.

`@extern(inject)` is for injections provided by the environment that is running
the CUE evaluator — for example, callbacks into a custom runtime environment
that embeds the CUE evaluator as a library. The injected values are provided
programmatically by the host application.

`@extern(go)` is specifically for Go code that is co-located with CUE source
and compiled into a plugin binary. Under the hood, the plugin binary uses the
inject machinery to provide the plugin values, but this is an implementation
detail — from the user's perspective, `@extern(go)` is a self-contained
mechanism that requires only the `cue plugin build` step.

A file should use one or the other, not both.

### LSP support

No explicit LSP support is needed for Go plugins beyond support for function
signatures when they come. The LSP server does not perform evaluation, so it
does not need to know about the actual Go code or the plugin binary. The
`_signatures_gen.cue` files generated by `cue plugin generate-go` provide
enough information for IDE features like completion and hover.

### Security and sandboxing

Deferred for later consideration. Plugin code runs as native Go code in the
same process with full system access, unlike sandboxed WASM. The explicit
`cue plugin build` step provides some user consent, but the silent re-exec
during `cue eval` is a trust escalation worth thinking about in the future.

In particular, cloning a repository and running `cue eval` could lead to
executing arbitrary native code (after the user runs `cue plugin build` as
instructed by the error message). There is currently no audit or review step
between discovering that plugins are needed and building/executing them.
Addressing this is important but is deliberately deferred to a follow-up design.

### Error handling

- If a plugin function returns a CUE value containing errors, those errors
  surface during CUE evaluation as one would expect.
- If the generated wrapper's reference map doesn't include a reference
  matching a `@go(...)` attribute, an error is produced at evaluation time
  indicating the missing plugin. For local references (`GoPackage == "."`),
  the error includes the CUE instance and module version; for external
  references, it includes the Go package path and module version.
- If the generated wrapper fails to compile (e.g. Go version incompatibility,
  missing dependencies), `cue plugin build` reports the Go compiler errors
  directly.

### Naming

`cue plugin build` is descriptive and doesn't conflict with existing commands.
The word "plugin" has some baggage from Go's `plugin` package, but in context
it's clear enough.

## Resolved decisions

### Generated code lifetime

The generated wrapper code lives in `cue.mod/plugin/` and is expected to be
committed to version control by most users. It functions as a lock file,
similar to committing `go.sum`. The manifest file (`manifest.json`) is also
committed alongside the generated code.

### Multiple plugin packages in one CUE module

Each injected value gets its own `@go(...)` attribute, in a file that has a
top-level `@extern(go)` attribute. There can be any number of these files in a
module; all share the same Go module (i.e. the single `go.mod` at or beneath
the CUE module root).

### Plugin binary location

The built binary lives in `CUE_CACHE_DIR/plugin/`, not inside `cue.mod/` — we
don't expect users to commit binaries to VCS. The binary name is
`cue_<hash>` (with `.exe` on Windows), where the hash is a SHA-256 of the
plugin `go.mod`, `main.go`, `GOARCH`, and `GOOS`. The `cue.mod/plugin/`
directory contains the generated wrapper source and manifest.

### Rebuilding triggers

For now, when `cue eval` (or similar) detects that a plugin binary is missing,
it produces an actionable error telling the user to run
`cue plugin build` rather than automatically triggering a rebuild. This
decision can easily be changed in the future.

## Discussion: deferred topics

### Plugin development workflow

During active plugin development, the requirement to run `cue plugin build`
after every Go code change adds friction to the edit-test-debug cycle. A future
improvement could include a `--watch` mode, an auto-rebuild option, or a
combined `cue plugin run` command. For now, this is accepted as a rough edge.

### Plugin testing

How users test their plugins in isolation is not yet addressed. Go code can be
unit-tested normally with `go test`. CUE-level integration testing of plugin
behavior (verifying that the CUE declarations and Go implementations work
together end-to-end) is a TODO. A `cue plugin test` command or similar may be
warranted in the future.

### Conflicting Go dependency versions across plugins

When multiple CUE module dependencies each bring their own Go dependencies,
Go's MVS will select the maximum version of each shared Go module. This could
in principle break a plugin that depends on the behavior of a specific older
version. There is no mitigation for this beyond what Go's module system already
provides — it is an inherent consequence of combining multiple Go dependency
graphs into a single binary.

### Using plugins without a main module

Currently `cue plugin build` requires a CUE main module (it looks for
`cue.mod/` to place generated code). This means one-off commands like
`cue export github.com/foo/bar@v0.1.2` cannot use plugins unless the user
creates a module first.

A possible future approach: `cue plugin build` could accept an explicit module
argument (e.g. `cue plugin build github.com/foo/bar@v0.1.2`) and generate
the wrapper code in the CUE cache directory rather than in `cue.mod/plugin/`.
The regular `cue` binary could then find the cached plugin binary based on the
module's dependency set. The details of cache location, invalidation, and
discovery for this case are TBD.
